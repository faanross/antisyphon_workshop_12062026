# mini-distillation

A small, self-contained TypeScript CLI that turns raw Zeek + Sysmon JSON logs
into **scored detection candidates**. It ships three detectors — network
**beacon**, **TLS anomaly**, and **PowerShell invocation anomaly** — and is built
to be read, run, and extended in an afternoon.

---

## 1. What this is

Threat hunting on raw telemetry is a needle/haystack problem: millions of conn
records, TLS handshakes, and process-creation events, almost all benign. The job
of the **distillation** stage is to boil that raw stream down to a short list of
*scored candidates* — entities that exhibit a measurable, suspicious **shape** —
so a human (or a later automated stage) only has to look at the interesting few.

```
  raw telemetry            distillation (THIS TOOL)         scored candidates
  ───────────────          ────────────────────────         ─────────────────
  zeek conn.log     ─┐     group by entity key
  zeek ssl.log       ├──▶  compute features            ──▶  BCN-… beacon_score=0.99
  sysmon EID 1/7    ─┘     weighted-sum / max → 0..1         TLS-… tls_score=0.95
                                                             PSI-… psi_score=0.95
```

This is a **faithful subset** of a larger 22-detector hunting system. The scorer
logic and weights here are ported verbatim from that system's
`beacon.ts`, `tls-anomaly.ts`, and `powershell-invocation-anomaly.ts` — only the
monorepo/infra coupling (databases, services, fleet baselines) was trimmed.

**This is the SCORING stage only.** It is deliberately *not* the whole pipeline:

- **Enrichment** — fleet-wide rarity tables, LOTS (Living-off-Trusted-Sites)
  classification, threat-intel feed correlation, GeoIP/ASN labels — happens
  *after* scoring and is **out of scope** here.
- **Attribution** — tying a network candidate (beacon) back to the process that
  made the connections — is a later stage and is **out of scope** here.

What you get is the pure, deterministic, math-only core: given filtered events,
produce scored candidates. No network, no database, no services.

## 2. Quickstart

Requires Node 18+ (uses `node:crypto`, ESM). Dev deps are only `tsx` +
`typescript` (+ `@types/node`).

```bash
npm install

# Run all three detectors on the bundled generic sample:
npx tsx src/distill.ts score all examples/events.sample.json

# Or one detector at a time, writing NDJSON to a chosen path:
npx tsx src/distill.ts score beacon     examples/events.sample.json --out beacon.ndjson
npx tsx src/distill.ts score tls        examples/events.sample.json --out tls.ndjson
npx tsx src/distill.ts score powershell examples/events.sample.json --out psi.ndjson
```

The CLI accepts either a **JSON array** of events or **NDJSON** (one event per
line). It prints a readable summary to stdout and writes candidates as NDJSON to
`--out` (default `candidates.ndjson`).

## 3. The three detectors

Each detector is a **pure function** from filtered events to scored candidates,
and produces a `0..1` score. Two scoring shapes are used: **weighted-sum** (beacon
— independent features that each add evidence) and **max-of-dimensions** (TLS,
PowerShell — independent signals where the *strongest* alone is enough).

### Beacon — `src/scorers/beacon.ts`

- **Reads:** Zeek `conn` events (`source: "zeek"`, `event_type: "conn"`).
- **Entity key:** `(src_ip, dest_ip, dest_port)`.
- **Measures:** how *metronomic* a conversation is. C2 beacons phone home on a
  timer with near-identical payloads; benign traffic is bursty and varied.
- **Features + weights** (weighted sum, the candidate must have ≥10 connections
  spanning ≥2 hours):

  | Feature | Weight | What it captures |
  |---|---:|---|
  | `regularity` | **0.30** | MAD-consistency of inter-arrival intervals (timer-like = 1.0) |
  | `bytes_out_consistency` | 0.15 | MAD-consistency of `orig_bytes` (fixed-size check-ins) |
  | `bytes_in_consistency` | 0.15 | MAD-consistency of `resp_bytes` (fixed-size responses) |
  | `duration_consistency` | 0.10 | MAD-consistency of connection `duration` |
  | `histogram_score` | 0.15 | `max(hourly-CV flatness, RITA-style bimodal score)` |
  | `consecutive_hours_norm` | 0.15 | longest run of consecutive active hours / 12 |

  `score = 0.30·regularity + 0.15·bytesOut + 0.15·bytesIn + 0.10·duration + 0.15·histogram + 0.15·consecutiveHoursNorm`

  Robust statistics (median, MAD) are used throughout so a few outlier
  connections don't wreck an otherwise clean beacon.
- **What a full fleet baseline adds:** nothing is *needed* here — beacon is fully
  self-contained. The real system layers **enrichment** on top: is this dest rare
  across the whole fleet? Is the JA3 a known C2 framework? Which process opened
  the connections (attribution)? Those raise/lower confidence but are separate
  stages.

### TLS anomaly — `src/scorers/tls-anomaly.ts`

- **Reads:** Zeek `ssl` events (`source: "zeek"`, `event_type: "ssl"`).
- **Entity key:** `(src_ip, dest_ip, dest_port)`, taking the max per dimension
  across all connections in the group.
- **Measures:** three independent suspicion dimensions; **composite =
  `max(cert, fingerprint, sni)`** (the strongest single signal wins, emitted when
  it clears `min_dimension_score = 0.30`):
  1. **Certificate anomaly** (additive, capped at 1.0): self-signed `+0.40`,
     expired `+0.20`, short validity `<7d` `+0.15`, long validity `>10y` `+0.15`,
     short serial `<4 bytes` `+0.10`. Self-signed **to an RFC1918 dest is
     excluded** (normal for internal services).
  2. **Fingerprint anomaly** (lookup): known-bad JA3+JA3S pair `0.95`, JA4X `0.95`,
     JA3 `0.90`, JA3S `0.80`.
  3. **SNI anomaly:** SNI/cert mismatch `0.70`, SNI present but connecting
     straight to a raw IP `0.60`, missing SNI entirely `0.50`.
- **What feeds add — and graceful degradation:** the **fingerprint dimension
  needs threat-intel feeds** (SSLBL-style known-bad JA3/JA4 lists). This tool
  ships a tiny *representative* feed in `data/feeds/`. **If the feeds are absent,
  the fingerprint dimension simply scores 0** and the candidate still fires on
  cert/SNI. In the full system these feeds are large, continuously updated, and
  scored with source-fidelity/freshness weighting (the **intel-match** stage).

### PowerShell invocation anomaly — `src/scorers/powershell-invocation-anomaly.ts`

- **Reads:** Sysmon `process_create` (EID 1) events, optionally correlated with
  `image_load` (EID 7) for the custom-host check (`source: "sysmon"`).
- **Entity:** the individual process-create event (`host`, `process_guid`).
- **Measures:** four independent suspicion dimensions; **composite =
  `max(rename, custom_host, parent, commandline)`** (emitted when it clears
  `min_dimension_score = 0.30`):
  1. **rename** — a non-PowerShell-named binary whose version metadata claims to
     *be* PowerShell (renamed `powershell.exe`) → up to `1.0`.
  2. **custom_host** — a non-standard process that loaded
     `System.Management.Automation.dll` (the PowerShell engine) → `lolbin`/`unknown`
     host `0.95`. Requires EID 7 telemetry; flagged `custom_host_uncheckable` if absent.
  3. **parent** — suspicious parent process via the taxonomy CSV: Office `0.95`,
     script hosts / LOLBins / web servers / browsers `0.85`, service hosts
     `0.20–0.50`, interactive shells `0.0`.
  4. **commandline** — offensive flag combinations, tiered: full offensive
     fingerprint or `-Version 2` downgrade `1.0`; ≥3 suspicious flags `0.90`;
     encoded + one other `0.75`; hidden + (encoded|noninteractive) `0.60`;
     encoded alone `0.40`. Uses a proper tokenizer (handles aliases like `-enc`,
     `-ep`, unicode dashes, quoting).
  - **Hard exclusions** (SCCM, Intune, Defender, Windows Update, etc.) and
    **vendor pairs** suppress known-benign management invocations before scoring.
- **What a full fleet baseline adds:** **no fleet rarity is needed** — this
  detector is self-contained on its config data. The real system adds
  per-environment prevalence (how common is this parent→child pair across the
  fleet?) and enrichment, again as separate downstream stages.

## 4. Extending it — add your own candidate

This is the point of the tool: the scorers are deliberately simple, pure
functions so you can write your own.

### The detector contract

> A detector is a **pure function** from a filtered slice of events to an array
> of scored candidates:
>
> ```ts
> function scoreXCandidates(events: XEvent[], config?: XConfig): XCandidate[]
> ```
>
> - **No side effects.** No network, no DB, no clock-dependence — same input,
>   same output (so candidate IDs stay deterministic).
> - **Output a `0..1` score** plus the sub-scores and the decisive raw evidence.
> - **Attach a deterministic id** via `assignDeterministicCandidateIds(prefix, …)`.

### Step by step

1. **Choose a telemetry source + entity key.** What log do you read (Zeek `conn`,
   `dns`, Sysmon EID 1, …)? What do you group by? The entity key *is* your unit
   of suspicion: `(src,dest,port)` for network, `process_guid` for endpoint.
2. **Define the signal / features.** Pick a small number of measurable features
   that capture the *shape* you care about. Favor robust stats (median/MAD over
   mean/stddev) so outliers don't dominate. The helpers in `src/stats.ts`
   (`mean`, `median`, `stddev`, `mad`, `madConsistency`, `round`) are there for you.
3. **Compute a `0..1` score.** Use **weighted-sum** when features are independent
   pieces of evidence that each add a little (like beacon), or **max-of-dimensions**
   when any single strong signal is sufficient (like TLS / PowerShell). Clamp to
   `[0,1]`.
4. **Attach evidence + a deterministic id.** Put the decisive raw values on the
   candidate (so a human can see *why*), include
   `evidence.constituent_event_ids`, and call
   `assignDeterministicCandidateIds('XYZ', candidates)`.
5. **Register it in the CLI dispatch.** In `src/distill.ts`: add your detector to
   the `Detector` union + `ALL_DETECTORS`, add a `filterXEvents` helper in
   `src/load.ts`, add a branch in `runDetector`, and a summary line in
   `summarizeCandidate`.

### Worked sketch — a `long_connection` detector

Goal: flag a single `(src,dest,port)` pair that holds connections open for a very
long *total* time (tunnels, persistent C2 channels). Three features: total
accumulated duration (normalized), connection count (normalized), and duration
consistency.

```ts
// src/scorers/long-connection.ts  (skeleton)
import type { ConnLogEvent } from '../types.js';
import { madConsistency, round } from '../stats.js';
import { assignDeterministicCandidateIds } from '../util.js';

export function scoreLongConnectionCandidates(events: ConnLogEvent[]) {
  const groups = new Map<string, ConnLogEvent[]>();
  for (const e of events) {
    const key = `${e.src_ip}|${e.dest_ip}|${e.dest_port}`;
    (groups.get(key) ?? groups.set(key, []).get(key)!).push(e);
  }

  const candidates = [];
  for (const [key, entries] of groups) {
    const durations = entries.map(e => e.duration ?? 0).filter(d => d > 0);
    if (durations.length < 2) continue;

    const total = durations.reduce((s, d) => s + d, 0);
    const durationNorm = Math.min(total / (8 * 3600), 1);          // 8h = "fully long"
    const countNorm    = Math.min(entries.length / 20, 1);          // 20 conns = saturated
    const consistency  = madConsistency(durations);                 // steady-state hold?

    const score = 0.55 * durationNorm + 0.20 * countNorm + 0.25 * consistency;
    if (score < 0.3) continue;

    const [src, dst, port] = key.split('|');
    candidates.push({
      candidate_id: '', type: 'long_connection',
      src_ip: src, dest_ip: dst, dest_port: parseInt(port),
      long_connection_score: round(score, 4),
      accumulated_duration_sec: round(total, 2),
      connection_count: entries.length,
      duration_consistency: round(consistency, 4),
      evidence: { constituent_event_ids: entries.map(e => e.id) },
    });
  }
  candidates.sort((a, b) => b.long_connection_score - a.long_connection_score);
  return assignDeterministicCandidateIds('LCN', candidates);
}
```

Then wire it into `distill.ts` exactly like the three built-in detectors.

### Calibration advice

- **Score how X-SHAPED the activity is, not whether it's malicious.** Benign
  software beacons too (update checkers, telemetry, NTP). A high beacon score
  means "this is very beacon-shaped," not "this is C2." Confidence about *malice*
  is what the later enrichment/attribution stages add.
- **Test against known-benign first.** Run your detector over a clean baseline
  capture and look at the score distribution. If legitimate software lands above
  your threshold, that's your false-positive budget — tighten features or add an
  allowlist (like the PowerShell hard-exclusions) rather than just raising the cut.
- **Weights are starting points.** The numbers above are sane defaults, not
  universal truths. Calibrate them on *your* data: which feature actually
  separates your true positives from the benign mass?
- **Keep it deterministic and robust.** Median/MAD over mean/stddev; clamp to
  `[0,1]`; never read the wall clock.

### Growing toward the full system

To approach the real 22-detector pipeline, layer these *on top of* the scoring
stage you have here:

- **Enrichment labels** — annotate each candidate with GeoIP/ASN, LOTS
  classification, and threat-intel hits (raise/lower confidence, don't re-score).
- **Fleet-wide rarity tables** — "how unusual is this dest / parent-child pair /
  JA3 across the entire estate?" turns a per-host shape into an estate-level signal.
- **Network → process attribution** — join a network candidate (beacon) to the
  Sysmon process that owned the socket, so a beacon stops being "an IP talks to an
  IP" and becomes "*this binary* beacons to *that* C2."

Those stages consume the candidates this tool emits — which is exactly why the
output here is structured, scored, and deterministically identified.
